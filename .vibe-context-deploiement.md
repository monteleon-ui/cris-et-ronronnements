## 📁 **Structure des Fichiers sur le Serveur**

Sur **PlanetHoster**, la structure typique pour une application Node.js avec Passenger est :

```
/home/
└── crisfahs/                          # Dossier utilisateur (exemple : crisfahs)
    ├── nodevenv/                      # Environnement Node.js (si utilisé)
    │   └── nocstorage/               # Dossier de l'application (exemple)
    │       ├── app.js                # Point d'entrée Express
    │       ├── package.json          # Dépendances npm
    │       ├── public/               # Fichiers statiques
    │       │   ├── assets/           # CSS, JS, images
    │       │   ├── robots.txt        # Règles pour les crawlers
    │       │   └── sitemap.xml        # Sitemap
    │       ├── server/               # Logique backend
    │       │   ├── middlewares/      # Middlewares Express
    │       │   ├── routes/           # Routes
    │       │   └── views/            # Templates EJS
    │       ├── logs/                 # Logs du serveur
    │       │   └── access.log        # Logs d'accès (Morgan)
    │       └── .env                  # Variables d'environnement (PRODUCTION)
    └── public_html/                  # Racine web (optionnel)
        └── .htaccess                 # Configuration Apache (si utilisé)
```

**Note** :
- **Passenger** utilise par défaut le dossier contenant `app.js` comme racine de l’application.
- Les fichiers statiques (`public/`) sont servis directement par Passenger.
- Le fichier `.env` **ne doit pas être commité** (déjà dans `.gitignore`).

## 🛠️ **Configuration pour PlanetHoster**

### **1. Configuration de Passenger**
Passenger est **préinstallé** sur PlanetHoster pour les hébergements mutualisés. Pour activer Node.js :

1. **Créer un fichier `passenger_wsgi.py`** (si nécessaire) :
   - Passenger détecte automatiquement les applications Node.js si un fichier `app.js` ou `server.js` est présent.
   - **Pas besoin** de configurer `passenger_wsgi.py` pour une application Node.js pure.

2. **Configurer le point d’entrée** :
   - Passenger utilise par défaut `app.js` comme point d’entrée.
   - Si vous utilisez un autre fichier (ex: `server.js`), spécifiez-le dans `package.json` :
     ```json
     "main": "server.js"
     ```

3. **Configurer le port** :
   - **Ne pas spécifier de port** dans `app.js` (Passenger gère automatiquement le port via la variable d’environnement `PORT`).
   - Exemple :
     ```javascript
     const PORT = process.env.PORT || 3000; // Passenger définit PORT
     app.listen(PORT, () => { ... });
     ```

### **2. Variables d’Environnement (`.env`)**

**Fichier `.env` en production** (à placer dans le dossier racine de l’application) :
```env
# Environnement
NODE_ENV=production

# Chemins des fichiers (à adapter selon la structure de PlanetHoster)
PUBLIC_PATH=/home/crisfahs/nodevenv/nocstorage/public
VIEWS_PATH=/home/crisfahs/nodevenv/nocstorage/server/views

# Port (géré par Passenger, pas besoin de le spécifier)
# PORT=3000

# Logging (optionnel)
LOG_LEVEL=info
```

**Exemple de `.env.example`** (pour le développement) :
```env
# Configuration de l'application Cris et Ronronnements
# Copiez ce fichier en .env et adaptez les valeurs

# Port du serveur (défaut: 3000)
PORT=3000

# Chemins des fichiers
PUBLIC_PATH=./public
VIEWS_PATH=./server/views

# Environnement (development, production)
NODE_ENV=development

# Configuration pour PlanetHoster (production)
# Passenger gère automatiquement le port et HTTPS
# PASSENGER_APP_ROOT=/home/crisfahs/nodevenv/nocstorage
```

### **3. Configuration de `package.json`**

**Scripts importants pour le déploiement** :
```json
{
  "scripts": {
    "start": "node app.js",
    "predeploy": "npm run clean && npm run build:all",
    "build:all": "npm run build:css && npm run build:js"
  }
}
```

**Dépendances** :
- Toutes les dépendances (`dependencies` et `devDependencies`) doivent être listées dans `package.json`.
- **Exécuter `npm install --production`** sur le serveur pour installer uniquement les dépendances de production.

### **4. Configuration de `.htaccess` (Optionnel)**

Si vous utilisez **Apache** (au lieu de Nginx) sur PlanetHoster, vous pouvez ajouter un fichier `.htaccess` dans `public_html/` pour :
- Forcer HTTPS
- Rediriger `www` vers le domaine nu
- Gérer les erreurs 404/500

**Exemple de `.htaccess`** :
```apache
# Forcer HTTPS
RewriteEngine On
RewriteCond %{HTTPS} off
RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]

# Rediriger www vers non-www
RewriteCond %{HTTP_HOST} ^www\.(.+)$ [NC]
RewriteRule ^ https://%1%{REQUEST_URI} [L,R=301]

# Gérer les erreurs
ErrorDocument 404 /404
ErrorDocument 500 /500
```

**Note** : Passenger gère déjà la plupart de ces redirections, donc `.htaccess` n’est **pas obligatoire**.

## 🚀 **Étapes de Déploiement**

### **1. Préparation Locale**

#### **a. Vérifier la configuration**
- **Node.js** : Vérifiez que votre version locale est **>= 20.0.0** (comme en production).
  ```bash
  node -v
  ```
- **Dépendances** : Exécutez `npm install` pour installer toutes les dépendances.
- **Variables d’environnement** : Copiez `.env.example` en `.env` et adaptez les valeurs pour le développement.

#### **b. Tester en local**
- Démarrez le serveur en mode développement :
  ```bash
  npm run dev
  ```
- Vérifiez que tout fonctionne :
  - Ouvrez [http://localhost:3000](http://localhost:3000) dans votre navigateur.
  - Testez toutes les routes (/, /bfk, /iufmm, etc.).
  - Vérifiez les pages d’erreur (404, 500).

#### **c. Exécuter les tests de linting**
```bash
npm run lint
npm run lint:fix  # Corrige les erreurs automatiquement
```

#### **d. Construire les assets**
```bash
npm run build:all
```
- Cela génère :
  - `public/assets/css/style.min.css` (CSS minifié)
  - `public/assets/js/config.min.js` (JS minifié)

#### **e. Valider le code**
```bash
npm run validate  # Exécute lint + build
```

  3. Naviguez jusqu’au dossier de votre application (ex: `/home/crisfahs/nodevenv/nocstorage/`).

  ```bash

  cd nodevenv/nocstorage
  ```

#### **b. Transférer les fichiers**

   ```bash

   git commit -m "Initial commit"
   ```
2. **Ajouter le remote PlanetHoster** :


   - **Hôte** : `sftp://crisfahs@sftp.planethoster.com`

   - **Mot de passe** : Votre mot de passe PlanetHoster
3. Transférez tous les fichiers (sauf `node_modules/` et `.git/`) vers `/home/crisfahs/nodevenv/nocstorage/`.

**Option 3 : Via `scp` (en ligne de commande)**

   ```bash

   npm install --production
   ```

**Via cPanel** :

## 🛑 **Dépannage**

### **Problèmes Courants et Solutions**

#### **1. L’application ne démarre pas**
- **Cause** : Fichier `app.js` manquant ou erreur de syntaxe.
- **Solution** :
  - Vérifiez que `app.js` existe dans le dossier racine.
  - Exécutez `npm run lint` pour détecter les erreurs de syntaxe.
  - Consultez les logs :
    ```bash
    tail -n 50 ~/nodevenv/nocstorage/logs/access.log
    ```

#### **2. Erreur 500 (Internal Server Error)**
- **Cause** : Erreur dans le code backend (ex: variable non définie, middleware manquant).
- **Solution** :
  - Vérifiez les logs pour voir l’erreur exacte.
  - Testez en local avec `NODE_ENV=production node app.js`.
  - Vérifiez que toutes les dépendances sont installées (`npm install --production`).

#### **3. Les assets statiques ne sont pas servis**
- **Cause** : Chemin incorrect pour `PUBLIC_PATH`.
- **Solution** :
  - Vérifiez que `PUBLIC_PATH` dans `.env` pointe vers le bon dossier.
  - Exemple :
    ```env
    PUBLIC_PATH=/home/crisfahs/nodevenv/nocstorage/public
    ```
  - Testez avec :
    ```bash
    ls -la /home/crisfahs/nodevenv/nocstorage/public/assets/css/
    ```

#### **4. Les variables d’environnement ne sont pas chargées**
- **Cause** : Fichier `.env` manquant ou mal placé.
- **Solution** :
  - Vérifiez que `.env` est dans le **même dossier que `app.js`**.
  - Vérifiez que `dotenv-safe` est installé (`npm list dotenv-safe`).
  - Testez en local :
    ```bash
    node -e "console.log(process.env.PUBLIC_PATH)"
    ```

#### **5. Problème de permissions**
- **Cause** : Passenger n’a pas les permissions pour accéder aux fichiers.
- **Solution** :
  - Vérifiez les permissions :
    ```bash
    ls -la ~/nodevenv/nocstorage/
    ```
  - Donnez les permissions nécessaires :
    ```bash
    chmod -R 755 ~/nodevenv/nocstorage/
    ```

#### **6. HTTPS ne fonctionne pas**
- **Cause** : Certificat SSL non configuré.
- **Solution** :
  - PlanetHoster **gère automatiquement** les certificats SSL via Let’s Encrypt.
  - Vérifiez que votre domaine pointe vers l’IP de PlanetHoster.
  - Attendez 24-48h pour la propagation du certificat.

#### **7. Les logs ne sont pas générés**
- **Cause** : Dossier `logs/` manquant ou permissions insuffisantes.
- **Solution** :
  - Créez le dossier `logs/` :
    ```bash
    mkdir -p ~/nodevenv/nocstorage/logs/
    ```
  - Donnez les permissions :
    ```bash
    chmod 755 ~/nodevenv/nocstorage/logs/
    ```

## 🔐 **Sécurité**

### **1. Bonnes Pratiques de Sécurité**
- **Ne jamais commiter** :
  - `.env` (contient des secrets)
  - `node_modules/` (dépendances)
  - `logs/` (logs sensibles)
- **Utiliser `dotenv-safe`** : Pour valider les variables d’environnement.
- **Limiter les permissions** :
  - `chmod 755` pour les dossiers.
  - `chmod 644` pour les fichiers.
- **Désactiver `X-Powered-By`** : Déjà fait dans `app.js`.
- **Utiliser CSP** : Déjà configuré avec Helmet.

### **2. Protection contre les Attaques**
- **Rate Limiting** : Déjà configuré (100 requêtes/15 min par IP).
- **CSP** : Déjà configuré avec nonce pour les scripts.
- **Helmet** : Déjà configuré pour les en-têtes HTTP sécurisés.
- **Validation des inputs** : Limite de taille pour les requêtes POST.

### **3. Surveillance des Logs**
- **Logs d’accès** : `logs/access.log` (généré par Morgan).
- **Logs d’erreurs** : Consultez les logs Passenger via cPanel.
- **Alertes** : Configurez des alertes pour les erreurs 500.
