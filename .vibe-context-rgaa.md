# Contexte RGAA : Cris et Ronronnements

## 🎯 **Objectif**
Ce document détaille les **mesures d'accessibilité** mises en place sur le site **Cris et Ronronnements** pour se conformer au **Référentiel Général d’Amélioration de l’Accessibilité (RGAA)**, version 4.1. L’objectif est de rendre le site **accessible à tous**, y compris aux personnes en situation de handicap (visuel, moteur, cognitif, etc.).

---

## 📜 **Contexte Légal**

### **Obligations en France**
- **Loi n° 2005-102** du 11 février 2005 : Obligation d’accessibilité pour les services publics.
- **Décret n° 2019-768** du 24 juillet 2019 : Application du RGAA 4.1 pour les services publics.
- **Directives européennes** : [UE 2016/2102](https://eur-lex.europa.eu/legal-content/FR/TXT/?uri=CELEX%3A32016L2102) (accessibilité des sites web des organismes publics).

**Note** : Bien que **Cris et Ronronnements** ne soit pas un service public, nous appliquons les bonnes pratiques du RGAA pour garantir une **expérience inclusive** à tous les utilisateurs.

## 📋 **Critères RGAA Appliqués**

Le RGAA 4.1 comprend **106 critères** répartis en **13 thèmes**. Voici ceux qui sont **implémentés** ou **en cours d’implémentation** sur le site.

## 🔹 **Thème 1 : Images**

### **Critère 1.1** : *Chaque image a-t-elle une alternative textuelle ?*
- **Statut** : ⚠️ **Partiellement implémenté**
- **Implémentation** :
  - Les images **statiques** (favicon, apple-touch-icon) n’ont pas d’alternative textuelle (car décoratives).
  - Les images **dynamiques** (si ajoutées via EJS) doivent inclure un attribut `alt` descriptif.
- **Exemple à suivre** :
  ```html
  <img src="/assets/images/illustration.jpg" alt="Illustration d'un questionnaire BFK" loading="lazy">
  ```
- **À faire** :
  - [ ] Ajouter des `alt` pour toutes les images dynamiques (ex: dans les pages BFK, IUFMM, Éditions SV).
  - [ ] Vérifier que les images décoratives ont un `alt` vide (`alt=""`).

### **Critère 1.2** : *Chaque image texte a-t-elle une alternative textuelle ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : Aucune image texte n’est utilisée (le texte est toujours en HTML).

### **Critère 1.3** : *Chaque image utilisée comme CAPTCHA ou comme image-test a-t-elle une alternative ?*
- **Statut** : ✅ **Non applicable** (pas de CAPTCHA ou image-test).

### **Critère 1.4** : *Chaque image porteuse d’information a-t-elle une alternative textuelle ?*
- **Statut** : ⚠️ **Partiellement implémenté**
- **À faire** : Vérifier que toutes les images porteuses d’information (ex: diagrammes, infographies) ont une alternative textuelle **détaillée**.

## 🔹 **Thème 3 : Couleurs**

### **Critère 3.1** : *Dans chaque page, l’information ne doit pas être donnée uniquement par la couleur. Utiliser un autre indicateur visuel en complément de la couleur.*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Les **liens** utilisent à la fois la couleur (`--link-color`) et le soulignement au survol (`text-decoration: underline`).
  - Les **boutons** utilisent des icônes ou du texte en complément de la couleur.
  - Les **alertes** (Bootstrap) utilisent des icônes + texte.

### **Critère 3.2** : *Dans chaque page, le contraste entre la couleur du texte et la couleur de son arrière-plan est-il suffisant ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Toutes les couleurs sont définies dans `public/assets/css/variables.css` avec un **contraste vérifié ≥ 4.5:1** sur fond blanc.
  - Exemples :
    - `--primary-color: #0056b3` → Contraste **7.5:1** sur blanc.
    - `--text-color: #212529` → Contraste **21:1** sur blanc.
    - `--secondary-color: #6c757d` → Contraste **4.5:1** sur blanc.
  - **Outil utilisé** : [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)
- **Exception** :
  - Le texte dans `.noscript-message` utilise `#664d00` sur `#fff3cd` → Contraste **5.5:1** (corrigé depuis `#856404` qui avait un contraste de 4.0:1).

### **Critère 3.3** : *Dans chaque page, les couleurs de premier plan et d’arrière-plan peuvent-elles être modifiées par l’utilisateur ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : Le site respecte les **préférences utilisateur** (ex: mode sombre via le navigateur ou le système d’exploitation).
- **À améliorer** :
  - [ ] Ajouter un **mode sombre** natif avec `prefers-color-scheme: dark`.

## 🔹 **Thème 5 : Tableaux**

### **Critère 5.1** : *Chaque tableau a-t-il un résumé ?*
- **Statut** : ⚠️ **Partiellement implémenté**
- **Implémentation** :
  - Aucun tableau n’est actuellement utilisé sur le site.
- **À faire** :
  - [ ] Si des tableaux sont ajoutés, utiliser `<caption>` pour fournir un résumé.
  - Exemple :
    ```html
    <table>
      <caption>Liste des formations disponibles dans le projet BFK</caption>
      <thead>
        <tr><th>Nom</th><th>Niveau</th></tr>
      </thead>
      <tbody>
    ```

### **Critère 5.3** : *Dans chaque tableau, chaque cellule de données est-elle associée à ses en-têtes ?*
- **Statut** : ⚠️ **Partiellement implémenté**
- **À faire** : Utiliser `scope="col"` ou `scope="row"` pour associer les cellules aux en-têtes.

## 🔹 **Thème 6 : Liens**

### **Critère 6.1** : *Chaque lien est-il explicite ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Tous les liens utilisent un **texte descriptif** (ex: "En savoir plus sur le projet BFK" au lieu de "Cliquez ici").
  - Les liens externes incluent `aria-label` pour plus de clarté.
  - Exemple :
    ```html
    <a href="/bfk" aria-label="En savoir plus sur le projet BFK">En savoir plus</a>
    ```

### **Critère 6.2** : *Chaque lien a-t-il un titre ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : Tous les liens ont un attribut `title` ou `aria-label` si nécessaire.

### **Critère 6.3** : *Dans chaque page, chaque lien est-il visible ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Les liens sont stylisés avec une couleur distincte (`--link-color: #0056b3`).
  - Les liens ont un **focus visible** (outline: 2px solid var(--primary-color)).
  - Les liens dans le footer sont séparés par des marges.

### **Critère 6.4** : *Dans chaque page, l’information indiquant la langue du lien est-elle disponible ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Les liens vers des pages en anglais utilisent `lang="en"`.
  - Exemple :
    ```html
    <a href="https://simonvitsonnet.com/contact/" lang="en" aria-label="Contacter Simon Vitsonnet (s'ouvre dans une nouvelle fenêtre)">Contact</a>
    ```

## 🔹 **Thème 7 : Scripts**

### **Critère 7.1** : *Chaque script est-il compatible avec les technologies d’assistance ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Aucun script ne bloque l’accès au contenu (le site fonctionne sans JavaScript).
  - Les scripts sont **non intrusifs** (ex: pas de redirections automatiques).
  - Le message `noscript` est affiché pour les utilisateurs sans JavaScript.

### **Critère 7.2** : *Chaque script a-t-il une alternative ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Le site est **entièrement fonctionnel sans JavaScript**.
  - Les fonctionnalités dynamiques (ex: menu déroulant Bootstrap) ont des alternatives (ex: liens directs).

### **Critère 7.3** : *Chaque script est-il contrôlable par le clavier ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Tous les éléments interactifs (liens, boutons, menus) sont **accessibles au clavier** (tabulation + Entrée/Espace).
  - Bootstrap 5 gère nativement la navigation au clavier.

### **Critère 7.4** : *Dans chaque page, le code source généré est-il valide selon le type de document déclaré ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Le code HTML est **valide HTML5** (vérifié avec [W3C Validator](https://validator.w3.org/)).
  - Le CSS est valide (vérifié avec [W3C CSS Validator](https://jigsaw.w3.org/css-validator/)).

### **Critère 12.2** : *Dans chaque page, le mécanisme de navigation principal est-il cohérent ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : La navigation est **identique** sur toutes les pages.

### **Critère 12.3** : *Dans chaque page, le mécanisme de navigation principal est-il accessible ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Navigation **accessible au clavier** (tabulation).
  - Menu déroulant géré par Bootstrap (accessible).
  - `aria-label` pour le menu principal.

### **Critère 12.4** : *Dans chaque page, le mécanisme de navigation principal est-il visible ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : La navigation est **toujours visible** (pas de menu caché par défaut).

### **Critère 12.5** : *Dans chaque page, y a-t-il un moyen de contourner les blocs de contenu répétitifs ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Lien **Skip-to-content** en haut de page :
    ```html
    <a href="#main-content" class="visually-hidden-focusable" style="...">Aller au contenu principal</a>
    ```
  - Le contenu principal a un `id="main-content"`.

### **Critère 12.6** : *Dans chaque page, les liens d’évitement ou d’accès rapide sont-ils pertinents ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : Le lien Skip-to-content pointe vers le contenu principal.

### **Critère 12.7** : *Dans chaque page, le focus est-il visible sur chaque lien d’évitement ou d’accès rapide ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Le lien Skip-to-content a un style de focus visible :
    ```css
    .visually-hidden-focusable:focus {
      position: static;
      width: auto;
      height: auto;
      overflow: visible;
      clip: auto;
    }
    ```

### **Critère 12.8** : *Dans chaque page, les liens de navigation sont-ils regroupés dans un conteneur dédié ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : La navigation est dans un `<nav>` dédié.

### **Critère 12.9** : *Dans chaque page, les liens de navigation sont-ils structurés ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** :
  - Utilisation de `<ul>` et `<li>` pour les menus.
  - Structure logique (Accueil → Projets → Sous-projets).

### **Critère 12.10** : *Dans chaque page, l’ordre de tabulation est-il cohérent ?*
- **Statut** : ✅ **Implémenté**
- **Implémentation** : L’ordre de tabulation suit l’ordre logique de la page.

## 🧪 **Outils pour Tester l’Accessibilité**

### **Outils Automatisés**
1. **[WAVE](https://wave.webaim.org/)** : Outil en ligne pour détecter les problèmes d’accessibilité.
2. **[axe DevTools](https://www.deque.com/axe/devtools/)** : Extension Chrome/Firefox pour auditer les pages.
3. **[Lighthouse](https://developer.chrome.com/docs/lighthouse/overview/)** : Intégré à Chrome DevTools (onglet "Accessibility").
4. **[Tanaguru](https://www.tanaguru.com/)** : Outil français pour auditer la conformité RGAA.
5. **[AccessiNum](https://accessinum.github.io/)** : Outil open-source pour évaluer l’accessibilité.

### **Outils Manuels**
1. **Navigation au clavier** :
   - Utiliser la touche `Tab` pour naviguer.
   - Vérifier que tous les éléments interactifs sont accessibles.
   - Vérifier que le focus est visible.
2. **Lecteur d’écran** :
   - [NVDA](https://www.nvaccess.org/) (Windows, gratuit).
   - [VoiceOver](https://support.apple.com/fr-fr/HT204408) (macOS/iOS, intégré).
   - [JAWS](https://www.freedomscientific.com/products/software/jaws/) (Windows, payant).
3. **Désactivation des styles** :
   - Désactiver le CSS dans le navigateur pour vérifier que le contenu reste lisible.
4. **Désactivation de JavaScript** :
   - Vérifier que le site fonctionne sans JavaScript.
5. **Mode haute contraste** :
   - Activer le mode haute contraste dans le système d’exploitation.

### **Vérificateurs de Contraste**
1. **[WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)** : Vérifier le contraste entre deux couleurs.
2. **[Coolors Contrast Checker](https://coolors.co/contrast-checker)** : Outil visuel pour tester les palettes.
3. **Extension Chrome** : [Color Contrast Analyzer](https://chrome.google.com/webstore/detail/color-contrast-analyzer/dagdlcijhfbmgkjokkjiagllieipbhjg).
